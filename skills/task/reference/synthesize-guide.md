# Synthesize Guide

Этот файл читается только в Фазе 3 — непосредственно перед записью task-файла.

Для каждого из 5 измерений: сначала напиши одно предложение рассуждения вслух («CoT-шаг»), потом формулируй секцию. Не пропускай CoT — он предотвращает галлюцинации на нетривиальных задачах.

---

## Измерение 1 — Intent Clarity

**Вопрос:** Два разных разработчика прочитают Task и сделают одно и то же?

**CoT-шаг перед написанием:** «Что именно нужно сделать — добавить, изменить, удалить, переместить, переписать? Если я заменю глагол на более конкретный, смысл изменится?»

**Правило:** Один глагол — одно действие. Заменяй размытые глаголы конкретными.

| Размыто | Конкретно |
|---|---|
| обработать | распарсить / записать в БД / отправить в очередь |
| улучшить | сократить время ответа с ~800ms до <200ms |
| исправить | вернуть 404 вместо 500 когда пользователь не найден |
| добавить поддержку | принимать `multipart/form-data` в endpoint `POST /upload` |
| рефакторить | разбить `UserService` (src/services/user.ts:1–340) на три класса |

**Формат секции Task:**
```
## Task

Добавить rate limiting в middleware `src/api/middleware/auth.ts` —
не более 100 запросов в минуту на IP, ответ 429 при превышении.
```

**Антипаттерн:**
```
## Task

Улучшить безопасность API endpoints.
```

## Измерение 2 — Scope Boundaries

**Вопрос:** Что точно входит в задачу, а что не входит — явно?

**CoT-шаг перед написанием:** «Что реализатор может случайно затронуть, считая это частью задачи? Что точно не нужно делать, даже если кажется логичным?»

**Правило:** Constraints — не банальщина. Каждый пункт отвечает на конкретный риск из findings фазы Investigate.

**Как заполнять Constraints:**

После Investigate у тебя есть список файлов и зависимостей. Для каждого файла вне прямого скоупа — реши: упомянуть как «не трогать» или промолчать. Упоминай если:
- файл логически связан и реализатор может решить его изменить
- файл хрупкий (много зависимостей, нет тестов)
- в нём есть похожий код который не нужно трогать

**Before / After:**

```
# Плохо — банальщина
## Constraints
- Не нарушать существующую функциональность
- Соблюдать code style проекта
- Писать чистый код
```

```
# Хорошо — конкретные риски из Investigate
## Constraints
- Не изменять интерфейс `IAuthMiddleware` (src/api/middleware/types.ts:12–28) —
  его используют 7 других middleware
- Не трогать `src/api/middleware/logging.ts` — там идентичный паттерн,
  но он специально отдельный (разные команды владеют файлами)
- Не добавлять зависимость на Redis — используй существующий in-memory store
  из `src/cache/memory-store.ts`
- rate limiting не применять к `/health` и `/metrics` endpoints
```

## Измерение 3 — Context Anchoring

**Вопрос:** Реализатор знает ровно столько, сколько нужно — не меньше и не больше?

**CoT-шаг перед написанием:** «Какие файлы из findings обязательно прочитает реализатор? Какие паттерны он должен повторить? Где точки входа и интеграции?»

**Правило:** Называй файлы с путями и номерами строк. Не «в модуле авторизации» — а `src/auth/middleware.ts:validateToken():89`.

**Структура хорошего Context:**

```
## Context

### Архитектура области

Запросы входят через `src/api/router.ts:42` → проходят цепочку middleware
в `src/api/middleware/index.ts:buildChain():15` → попадают в handlers.

Текущая цепочка (src/api/middleware/index.ts:15–34):
1. cors() — src/api/middleware/cors.ts
2. bodyParser() — src/api/middleware/body.ts
3. auth() — src/api/middleware/auth.ts  ← точка изменения
4. requestLogger() — src/api/middleware/logging.ts

### Паттерны

Все middleware следуют интерфейсу `IMiddleware` (src/api/middleware/types.ts:8):
возвращают `(req, res, next) => void`, бросают `AppError` при отказе.

Пример похожей логики — throttle для WebSocket в `src/ws/throttle.ts:24–67`:
использует sliding window через `src/cache/memory-store.ts`.

### Тесты

Тесты middleware лежат в `src/api/middleware/__tests__/`.
Текущее покрытие auth.ts — 84% (src/api/middleware/__tests__/auth.test.ts).
```

**Антипаттерн:**
```
## Context

Проект использует Express.js с middleware-архитектурой.
Авторизация реализована в модуле middleware.
Есть кэш для хранения данных.
```

## Измерение 4 — Acceptance Criteria

**Вопрос:** Реализатор может проверить каждый критерий не запуская весь проект?

**CoT-шаг перед написанием:** «Что именно должно работать после реализации? Какую команду я запущу чтобы убедиться? Что произойдёт в граничных случаях?»

**Правило:** Каждый пункт Verification — конкретная команда с ожидаемым результатом или наблюдаемое поведение. Добавляй edge cases которые нашёл в Investigate.

**Before / After:**

```
# Плохо — не проверяемо
## Verification

- Rate limiting работает корректно
- Существующие тесты проходят
- Нет регрессий
```

```
# Хорошо — команды и поведение
## Verification

- `npm test src/api/middleware/__tests__/auth.test.ts` — все тесты зелёные
- `npm test` — общий suite без новых failures
- curl 101 раз подряд → первые 100 возвращают 200, 101-й возвращает 429
  с телом `{"error": "Too Many Requests", "retryAfter": 60}`
- curl с разных IP → лимиты независимы (IP A не влияет на счётчик IP B)
- `GET /health` при исчерпанном лимите → 200 (rate limiting не применяется)
- Перезапуск сервера → счётчики сбрасываются (in-memory, персистентность не нужна)
```

**Откуда брать edge cases:** из findings task-explorer. Если агент нашёл `/health` endpoint или несколько IP в тестах — это сигнал что это важный case.

## Измерение 5 — Output Format

**Вопрос:** Реализатор знает как именно предоставить результат?

**CoT-шаг перед написанием:** «Как реализатор должен двигаться — план сначала или сразу код? Нужны ли тесты? Один подход или несколько вариантов?»

**Правило:** Встрой инструкцию в конец секции Task или в начало Requirements. Выбирай режим исходя из сложности:

| Сложность | Режим |
|---|---|
| trivial / simple | «Реализуй напрямую» |
| medium | «Сначала напиши план (файлы + порядок), жди подтверждения, потом код» |
| complex | «Предложи 2 подхода с trade-offs, жди выбора, потом план, потом код» |

**Примеры формулировок:**

```
# trivial
## Task

Исправить опечатку в сообщении ошибки в `src/auth/errors.ts:23`.
Реализуй напрямую.
```

```
# medium
## Task

Добавить rate limiting в `src/api/middleware/auth.ts`.
Сначала напиши план: какие файлы создаёшь/меняешь и в каком порядке.
Жди подтверждения перед реализацией.
```

```
# complex
## Task

Мигрировать систему авторизации с JWT на session-based.
Предложи 2 подхода с trade-offs (backward compatibility vs чистота архитектуры).
После выбора — план файлов и порядок, жди подтверждения, потом код.
```

## Классификация сложности

Определи сложность до написания Task — она влияет на Output Format и на уровень детализации Context.

| Сложность | Признаки | Типичный объём Context |
|---|---|---|
| **trivial** | 1 файл, 1–5 строк, нет зависимостей | 2–3 строки |
| **simple** | 1–2 файла, понятный скоуп, есть тесты | 5–15 строк |
| **medium** | 3–7 файлов, нужно понять связи, возможны регрессии | 15–40 строк |
| **complex** | затрагивает архитектуру / несколько слоёв / нет тестов / публичный API | 40+ строк, схема data flow |

**Признаки сложнее чем кажется:**
- нет тестов на затронутую область → medium минимум
- изменение публичного интерфейса (экспортируемые типы, API endpoint) → +1 уровень
- несколько команд владеют файлами → явно обозначь в Constraints
- task-architect нашёл несколько несовместимых паттернов → complex, нужен Output Format «2 подхода»

## Уточняющие вопросы

Генерируй **ровно 5 вопросов, максимум 7**. Только если ответ изменит реализацию.

**Тест на нужность вопроса:** «Если я угадаю ответ — реализатор сделает неправильно?» Если да — вопрос нужен. Если нет — убери.

**Формат:**
```
1. **Вопрос в одном предложении?**
   - [ ] Вариант A — что это означает для реализации
   - [ ] Вариант B — что это означает для реализации
   - [ ] Свой вариант: ___
```

**Примеры хороших вопросов** (про реализацию, не про постановку):

```
1. **Где хранить счётчики rate limiting?**
   - [ ] In-memory (текущий паттерн в проекте) — сбрасывается при рестарте,
         не работает при горизонтальном масштабировании
   - [ ] Redis — персистентно, нужна новая зависимость
   - [ ] Свой вариант: ___

2. **Как считать лимит — по IP или по пользователю?**
   - [ ] По IP — проще, но ломает shared NAT
   - [ ] По user ID из JWT — точнее, требует парсинга токена до rate check
   - [ ] Оба — IP для анонимов, user ID для авторизованных
   - [ ] Свой вариант: ___
```

**Антипаттерн — вопросы не про реализацию:**
```
# Плохо
1. Что именно нужно сделать?
2. Есть ли дедлайн?
3. Кто будет ревьювить?
```
