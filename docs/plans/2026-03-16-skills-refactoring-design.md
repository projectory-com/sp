# Рефакторинг цепочки /task, /plan, /do — дизайн

Дата: 2026-03-16
Scope: критичные + важные findings из deep review
Подход: три параллельных рефакторинга по скиллам + сквозной проход

## Решения принятые при brainstorming

- **Scope:** критичные + важные (~14 проблем). Улучшения (auto-discovery, scope guard, actionable notification) -- за пределами scope.
- **Агенты:** объединить explorer + designer/architect в одного агента в /task и /plan. Меньше потерь контекста при передаче findings.
- **Parallel/agent-team:** оставить в /plan с V1-пометкой. /do игнорирует, но план будет готов к параллельному исполнению.
- **Task-файл:** read-only для /do. Вся информация о выполнении -- в report-файле и TodoWrite.
- **Приоритет:** равномерно по скиллам, организовать по блокам.

### 1.2. Перенести TASK_TYPE в Фазу 3

- **Почему:** /plan (строка 39) ожидает TASK_TYPE. Сейчас контракт нарушен.

### 1.4. Объединить task-explorer + task-architect

- **Следствие:** SKILL.md Фаза 2 упрощается: один Agent-вызов вместо двух с промежуточной передачей.

- **Почему:** инструкция "прочитай в Фазе 3" достаточна.

### 1.6. Подсказка следующего шага
- **Что:** в Фазу 4 (Write) добавить: "Следующий шаг: `/sp:plan <путь>`".
- **Почему:** /plan уже подсказывает /do, /task -- нет.

### 1.7. Убрать дублирование
- **Что:** промты агентов из SKILL.md (строки 77-108) заменить на ссылки на agents/. Правило про вопросы (строка 138) убрать из SKILL.md, оставить в synthesize-guide.

## Блок 3: /do

### 3.1. Убрать двойную верификацию
- **Что:** в sub-agents flow убрать повторный verify оркестратором после DONE. Доверять sub-agent если DONE + коммит сделан.
- **Почему:** между коммитом sub-agent и проверкой оркестратора ничего не меняется.

### 3.2. Убрать дублирование коммита
- **Что:** убрать шаг 4 ("Коммит") из sub-agents flow. В шаге 3: "коммит если sub-agent не сделал".
- **Почему:** коммит упоминается трижды (шаг 3, шаг 4, task-executor).

### 3.3. Format перед Report
- **Что:** переставить порядок Фазы 7: Format -> Report -> Notification.
- **Почему:** коммит format не попадает в report при текущем порядке.

### 3.4. Guard на пустой файл-лист
- **Что:** если все tasks BLOCKED/SKIPPED (изменённых файлов ноль) -- пропустить Фазы 3 (Simplify), 4 (Cleanup), 6 (Document). Перейти к Report со статусом failed.
- **Почему:** Simplify/Cleanup на пустом входе = пустые коммиты и потраченные токены.

### 3.5. Упростить парсинг Фазы 1
- **Что:**
  - Убрать PARALLEL из списка полей (V1: всё sequential)
  - Убрать MODE (V1: trivial/simple -> inline, medium/complex -> sub-agents)
  - Объединить EXECUTION_ORDER с TASKS: "в порядке из Execution/Order или появления"
  - Итого: 5 полей вместо 8 (SLUG, COMPLEXITY, TASKS[], CONSTRAINTS, VERIFICATION)
- **Почему:** парсинг неиспользуемых полей = шум в инструкциях.

### 3.6. Task-файл read-only
- **Что:** убрать шаг 6 Фазы 1 ("записать todo list в task-файл") и обновление секции "Выполнение" из Фазы 7.
- **Почему:** task-файл -- входной документ /task. Прогресс -- в TodoWrite, итог -- в report.

### 3.7. Убрать дублирование commit convention
- **Что:** убрать конкретные commit messages из описания фаз. Оставить: "Коммит по конвенции из reference/commit-convention.md".
- **Почему:** два источника правды.

### 3.8. Проверка неотвеченных вопросов из плана
- **Что:** в Фазу 1 (Parse) добавить: "Если план содержит неотвеченные уточняющие вопросы -- не продолжать."
- **Почему:** вопросы могут влиять на реализацию.

## Блок 4: Сквозной проход

После блоков 1-3 проверить контракты:
1. /task выводит `**Тип:**` -> /plan его читает в TASK_TYPE
2. /task подсказывает `/sp:plan` -> /plan подсказывает `/sp:do`
3. /plan записывает "Уточняющие вопросы" -> /do проверяет чекбоксы
4. Согласованность формата task-файла между шаблоном /task и парсером /plan
5. Согласованность формата plan-файла между plan-format.md и парсером /do
