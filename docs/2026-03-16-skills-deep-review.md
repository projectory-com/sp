- **Auto-detection пути `docs/ai/`** -- если в проекте другая структура, не хардкодить.
- **Вывод вопросов в чат** после записи файла -- пользователь может ответить не открывая файл.
- **Slug-генерация** -- сократить 6 строк до 2.

### 4. Логичность и последовательность

Общий порядок Parse -> Execute -> Simplify -> Cleanup -> Validate -> Document -> Report **логичен**.

- Simplify перед Cleanup -- правильно.
- Document после Validate -- правильно.
- Но Simplify + Cleanup -- два sub-agent'а на смежных задачах. Можно объединить в будущем.

Verify failure handling неконсистентен: inline -- "одна попытка", sub-agents -- retry за NEEDS_CONTEXT + retry за verify = 2 re-dispatch без ограничения. Добавить лимит.

### 5. Что стоит добавить

- **Pre-flight checks** в Фазу 1: чистый git, существование файлов, отсутствие неотвеченных вопросов.
- **Error recovery** в Правила: обработка таймаутов, merge conflicts, disk errors.
- **Scope guard**: проверка что sub-agent не вышел за пределы task.Files.

### 6. Где упростить

- **Auto-detect plan-файла** -- найти последний `docs/ai/**/*-plan.md`.
- **Парсинг**: 5 полей вместо 8 (убрать PARALLEL, MODE, EXECUTION_ORDER).
- **Actionable notification**: подсказать следующий шаг (review + PR, или manual fix).

### Что сделано хорошо

- Pipeline из 7 фаз с чётким разделением
- Status protocol: 4 статуса покрывают основные случаи
- "При BLOCKED продолжать с независимыми tasks" -- правильное поведение
- Agent промты лаконичны и фокусированы
- Принцип "запустил и ушёл" -- правильная ментальная модель
