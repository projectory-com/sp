# Коммит с привязкой к тикету

Создать атомарный коммит: определить ticket ID, сформировать conventional commit message, зафиксировать изменения.

---

## Шаг 1 — Сбор контекста

Выполни параллельно:

```bash
git status
```

```bash
git diff HEAD
```

```bash
git diff --stat HEAD
```

```bash
git ls-files --others --exclude-standard
```

Если изменений нет (ни staged, ни unstaged, ни untracked) — сообщи пользователю и останови.

---

## Шаг 2 — Определение ticket ID

Каскад приоритетов:

### 2a. Из аргументов

`$ARGUMENTS` может содержать ticket ID или URL. Извлекай по таблице:

| Вход                                      | Ticket ID  |
| ----------------------------------------- | ---------- |
| `86` или `#86`                            | `#86`      |
| `R2-50`                                   | `R2-50`    |
| `https://github.com/.../issues/86`        | `#86`      |
| URL YouTrack содержащий `/issue/PROJ-123` | `PROJ-123` |

Если `$ARGUMENTS` пуст или ticket отсутствует — перейди к 2b.

### 2b. Из имени ветки

```bash
git branch --show-current
```

Паттерны:

| Имя ветки               | Regex          | Ticket ID  |
| ----------------------- | -------------- | ---------- |
| `86-feature-name`       | `^(\d+)-`      | `#86`      |
| `feature/86-name`       | `\/(\d+)-`     | `#86`      |
| `R2-50-feature`         | `(R\d+-\d+)`   | `R2-50`    |
| `fix/R2-50-name`        | `(R\d+-\d+)`   | `R2-50`    |
| `PROJ-123-feature`      | `([A-Z]+-\d+)` | `PROJ-123` |
| `main`, `develop`, etc. | —              | не найден  |

Если ticket найден — используй его. Если нет — перейди к 2c.

### 2c. Спросить пользователя

Спроси через AskUserQuestion с вариантами:

- **Без тикета** — закоммитить без привязки
- **Ввести номер** — пользователь вводит свой ticket ID

---

## Шаг 3 — Стейджинг

Добавь все изменённые и новые файлы. **НЕ добавляй:**

- `.env`, `.env.*` — секреты
- Файлы с credentials, tokens, keys
- Большие бинарные файлы

Если среди untracked есть такие файлы — предупреди пользователя перед добавлением.

Добавляй конкретные файлы по именам (не `git add -A`).

---

## Шаг 4 — Commit message

Формат:

```
[ticket] <type>(<scope>): <описание>
```

**ticket** — ставь первым. `#86` для GitHub, `R2-50` / `PROJ-123` для YouTrack. Без тикета — опусти.

**type** — определи по характеру изменений:

| Тип        | Когда                               |
| ---------- | ----------------------------------- |
| `feat`     | Новая функциональность              |
| `fix`      | Исправление бага                    |
| `refactor` | Рефакторинг без изменения поведения |
| `docs`     | Только документация                 |
| `test`     | Только тесты                        |
| `chore`    | Сборка, CI, зависимости, конфиги    |
| `perf`     | Оптимизация производительности      |

**scope** — опционально, основная область изменений.

**описание** — одно предложение, активный залог, на языке проекта.

Примеры:

- `#86 feat(pages): добавить страницу блэкджека`
- `R2-50 fix: исправить сохранение user ID в БД`
- `chore: обновить зависимости`

---

## Шаг 5 — Коммит

```bash
git commit -m "<message>"
```

Без `Co-Authored-By`, `Signed-off-by` и любых trailer lines.

После коммита покажи результат: хэш, сообщение, список файлов.

---

## Правила

- Один коммит — одно логическое изменение
- Ticket ID всегда первым в сообщении (если есть)
- НЕ используй `wip`, `temp`, `misc`
- НЕ коммить секреты и credentials
- Сообщение коммита — краткое, конкретное, в активном залоге
